# wazuh-mcp M4c retro (v0.6.0-m4c)

Ship date: 2026-04-27. Successor to `2026-04-24-m4b-retro.md`.

## Headline

Shipped per-tenant policy resolution + write-surface completion + multi-agent AR + `confirm_required` cleanup. Closed the multi-tenant policy-bleed gap carried over from M4b. 17 plan tasks across 3 phases. **15 subagent dispatches** (14 implementer + 1 combined Tier-A reviewer); **zero fix-after-review cycles**. Unit test count 429 → 483 (+54). Three new integration tests added (run nightly on amd64 CI). Single tag `v0.6.0-m4c`. Behavior delta-free for single-tenant deployments; multi-tenant HTTP deployments now resolve `role_tool_allowlist`, `write_allowlist`, and `active_response_allowlist` correctly per session tenant.

## What went well

- **Plan-detail investment paid off again.** M4b's lesson held: concrete Pydantic fields, test bodies, and handler code spelled out inline produced zero fix-after-review cycles, matching M4b and improving on M4a's 3.
- **Tier-A spot-check approach validated.** One full combined-reviewer dispatch on T3 (resolver primitive). Five Tier-A spot-checks on T6/T7/T8/T9/T12 (composition tasks downstream of T3). Zero issues slipped through to the suite. Reviewer verdict on T3 was "ship as-is" with only minor non-blocking test-symmetry suggestions — the implementer's adherence to the spec snippet was tight.
- **Resolver factor extracted cleanly.** `src/wazuh_mcp/rbac/resolver.py` came out as a small focused module (~80 lines, three factory functions, two module constants). The `server.py` wiring update was mechanical: replace inline `_rbac_policy(session)` closures with `make_*` factory output, plumb new kwargs through `_register_everything`. Both stdio and HTTP modes share the wiring via `SingleTenantRegistry` / `YamlTenantRegistry` symmetry.
- **Multi-agent AR refactor was trouble-free.** Single subagent dispatch (T9) handled the breaking schema change cleanly: refactored `IsolateAgentArgs`/`RunActiveResponseArgs`, added `FailedAgent` model + `_AR_AGENTS_MAX = 50`, added `_extract_failed_items` helper, refactored `ServerApiClient.isolate_agent`/`run_active_response` to comma-join, migrated 3 call sites in `test_server_api_writes.py`, 6 call sites in `test_write_tools.py`, schema field in `test_instrumented_tool.py:477`, and 3 FastMCP payloads in `test_m4b_writes.py`. No fix cycle.
- **`cluster.status` proved a useful pairing with `restart_manager`.** Operators get a single MCP-callable readiness check; the M4b out-of-band restart pattern is now optional.
- **Phase-1 "behavior delta-free" approach worked.** The plan landed `tenant_cfg=`-registration filtering and resolver-call-time filtering in parallel during phase 1; phase 2 (T8) removed the registration filter. Single-tenant deployments saw zero behavior change after T1-T7 — only the M4b → M4c transition introduces the operator-visible delta documented in `m4c-multi-tenant.md` §3.

## What surprised us

- **Plan-time fixture drift was the dominant cost.** Six tasks (T2/T3/T4/T5/T8/T10) had test fixtures referencing fields/imports that didn't exist on the actual codebase: `TenantConfig.wazuh_url`, `TenantConfig.api_key_default_role`, `TenantConfig.wazuh_user_claim`-positional, `set_current_session` from `observability.contextvar` (real path: `transport.session_ctx`), `InProcessRateLimiter` from `observability.ratelimit` (real: `rate_limit.limiter`), `SecretValue` from `security.secret` (real: `secrets.value`). Each was caught pre-dispatch by ~30s of `grep` on the actual signatures. Aggregate controller overhead: ~3 minutes/task × 6 tasks = 18 min. Aggregate cost-avoided: ~6 implementer fix cycles (~10 min each) = ~60 min. Net win ~42 min, but the lesson is clear: **plan author should grep all TenantConfig / pool-class / fixture-helper signatures before authoring test fixtures**. The M4b retro flagged this, but the M4c plan repeated the pattern. Worth elevating to the methodology memory.
- **FastMCP wraps `WazuhError` into `ToolError`.** The T8 plan asserted `pytest.raises(WazuhError)` for the call-time write-allowlist deny — actual behavior is `mcp_app.call_tool` wraps internal exceptions in `ToolError(...) from e`. The implementer adapted correctly: catch `ToolError`, assert `__cause__` is `WazuhError` with `code == "forbidden"`. Underlying behavior still pinned. The unit-test-via-FastMCP shape needs this adapter.
- **`ServerApiClient` isn't an async context manager.** The T10 plan sketched `async with _client() as c:`; actual M4b precedent (`tests/unit/test_server_api_writes.py:14-43`) uses an async generator fixture with `try/yield/finally + await c.aclose()`. Caught pre-dispatch.
- **`requires_manager` is a pytest mark, not an importable name.** T14 plan had `from tests.integration.conftest import requires_manager`; actual usage is `pytestmark = [pytest.mark.integration, pytest.mark.requires_manager]`. Caught pre-dispatch.
- **`_mcp_session(url, token)` lives in `tests/integration/test_m4b_writes.py:186`, not `conftest.py`.** The M4b retro recorded this same finding ("inline-per-file fixture precedent"). T14 plan re-imported it from conftest. Caught pre-dispatch.
- **Pydantic Literal mismatches don't trigger ty errors at call sites.** The T11 plan added `# ty: ignore[invalid-argument-type]` on `RestartManagerArgs(scope="rolling", ...)` and `RestartManagerArgs(confirm=False)` test calls; ty doesn't flag these (Pydantic call sites don't go through ty's Literal-checking path). The implementer correctly removed the unnecessary ignores. Worth knowing for future Pydantic-validation tests.
- **LSP indexing lag persists as a noise floor.** Across all 14 implementation dispatches, the runtime LSP emitted `unresolved-import` and `unresolved-attribute` errors after every new file creation or symbol addition; CLI `ty check` stayed green throughout. The methodology memory already documents this; it remained accurate this milestone.
- **Implementer-flagged observation worth noting.** T3 implementer flagged that `make_write_allowlist` returns `[]` (not `None`) on unknown-tenant — this is fail-closed-by-design and aligns with `_check_write_allowed` (`None` → no filter; `[]` → empty allowlist denies). The reviewer confirmed alignment by reading the downstream call site. This is the kind of "tier-A framing invites proactive security analysis" pattern M4b recorded.

## Carry-forward to M4d / M5

### Deferred from M4c
- **Group-target `run_active_response`** (`group_id: str` arg). TOCTOU + post-resolution cap behavior need their own design pass. M4c spec §11.
- **Per-tenant rate-limiter / audit-sink fan-out.** Today's `_build_sinks(http_cfg.tenant)` and `InProcessRateLimiter(default=tenant.rate_limit)` are still primary-tenant captured at server-build time. Multi-tenant per-tenant rate limiting and per-tenant sink routing is the natural "M4d" or "M5 multi-tenant infra completion" anchor.
- **MCP elicitation activation of `confirm_required`.** Gated on SDK shipping native elicitation support. Cleanup landed in M4c (`confirm_required` removed from `SAFE_CODES`); re-add when SDK lands the feature.
- **Formal toolset SDK wiring.** Gated on `mcp` SDK shipping native toolset client-enablement. M4b T1 probe pinned `mcp 1.27.0` has no native support; revisit when SDK ships.
- **`WazuhError.scope` field for rate-limit metrics.** M4a nit, still deferred.
- **`cancelled` outcome vocabulary.** M4a addition, still deferred.
- **Vault integration tests.** Still unit-only via `hvac.Client` mocks.

### M5 ship-gate items still outstanding
- Eval harness (~40 NL prompt → tool-call pairs; ≥90% selection accuracy gate).
- Wazuh LTS + latest matrix in CI.
- Cross-tenant leak test suite (session pinned to tenant A runs every read AND write tool; assert no tenant B data/resource touched). M4c per-tenant resolver primitive is now the prerequisite.
- Multi-manager integration fixture (two distinct Wazuh clusters in CI).
- `pip-audit` / `safety` in CI; secret-leak scanner on integration log corpus.
- Helm chart for k8s deploy.

### Methodology refinements for M4d
1. **Plan-time signature grep is mandatory.** Plan author runs `grep -rn "class TenantConfig\|class Session\|class .*RateLimiter\|class SecretValue\|^def _mcp_session\|^def set_current_session"` and pastes verified import paths into every test fixture before saving the plan. Costs ~5 minutes; saves 30+ minutes of dispatch-time corrections per milestone.
2. **FastMCP exception-wrapping is the call-time integration test pattern.** When asserting on tool call errors via `mcp_app.call_tool`, expect `ToolError(...) from e`; assert against `__cause__` for the underlying `WazuhError`. M4b precedent had this implicit; M4c plan got it wrong; methodology memory should record it.
3. **Tier-A spot-check is the default.** Two milestones running with zero fix cycles under spot-check + 1 full review. M4d should default to spot-check unless a task introduces a genuinely novel security primitive.
4. **`DONE_WITH_CONCERNS` is healthy signal.** Both DONE_WITH_CONCERNS reports this milestone (T8, T11) were implementer judgments correcting plan drift — exactly the kind of feedback the methodology aims to surface. Treat them as plan-correction insight, not as implementation-quality concerns.
