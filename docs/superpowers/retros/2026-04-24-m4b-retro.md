## Outcome

M4b shipped. **428 unit tests + 26 integration tests** (6 new M4b, 20 M4a). +34 unit tests for M4b across the session (394 at M4b start → 428 at ship). Ruff + ruff format + ty + pytest all clean. 10 tasks, 9 implementer dispatches (T1 + T2 batched as one), zero tier-A review-fix cycles — significantly cleaner than M4a (which had 3 review-fix cycles across tier-A tasks).

Landed surfaces:

- **Seven `write.*` tools** under a dedicated namespace, each with `confirm: Literal[True]` Args gating at Pydantic parse time, double-audit (pre-call `write.requested` + post-call completion), `run_as=session.wazuh_user` attribution to Wazuh's own audit log. Full list: `write.isolate_agent`, `write.restart_agent`, `write.add_agent_to_group`, `write.remove_agent_from_group`, `write.create_rule`, `write.update_rule`, `write.run_active_response`.
- **`@instrumented_tool` extension** — single conditional emit inside the span block (`if tool_name.startswith("write."):`) before handler execution. Non-write tools retain the M4a single-event contract; pinned by a regression test.
- **`RuleDefinition` + `render_rule_xml`** pure XML renderer. Pydantic `@field_validator` on `regex` (`re.compile` gate), plus a `@model_validator(mode="after")` that scans every user-controlled string for XML-1.0-illegal codepoints (C0 controls, unpaired surrogates, `U+FFFE`/`U+FFFF`) — a real security gap the plan didn't anticipate that the T3 implementer surfaced and closed.
- **Two-layer allowlist** — registration-time `TenantConfig.write_allowlist: list[str] | None` (None → all writes register, empty → none register, named list → only listed tools register) + call-time RBAC inherited from M4a. `active_response_allowlist: list[str]` with deny-all-by-default semantics for `write.run_active_response`.
- **`ServerApiClient` write methods.** Six Wazuh-specific methods (`isolate_agent`, `restart_agent`, `add_agent_to_group`, `remove_agent_from_group`, `upload_rule_file`, `run_active_response`) plus generic `put`/`delete`/`put_raw` helpers, all with `run_as` query-param plumbing.
- **`confirm_required` in `SAFE_CODES`** — declared as a defensive post-parse code; in practice, missing `confirm=True` fails Pydantic's `Literal[True]` check and surfaces as `parse_error` (caught by T9 docs implementer; documented honestly).
- **Toolset SDK probe** — `mcp==1.27.0` has no native toolset client-enablement surface. `meta={"toolset": "writes"}` stays as the descriptive placeholder; enforcement lives at the `_register_everything` registration-time filter. Decision documented in `docs/superpowers/notes/2026-04-24-toolset-sdk-probe.md`.
- **Integration tests.** 6 new write-tool roundtrips in `tests/integration/test_m4b_writes.py`: isolate→restart happy path, add→remove group, create_rule upload, `run_active_response` allowlist rejection, `confirm` missing rejection, double-audit indexer-sink roundtrip. Inline server subprocess pattern matching the M4a precedent — no shared conftest changes.
- **Operator docs** — `docs/deploy/m4b-writes.md` (217 lines) with 9 sections covering setup, both allowlists, custom `responder` role example, run_as verification via `/var/ossec/logs/api.log`, Claude-ops confirm-flow expectations, rule-file lifecycle, audit-shape saved searches.

## What went well

- **The write-flow template composed cleanly.** `confirm: Literal[True]` + `@instrumented_tool` + two-layer allowlist + `run_as=session.wazuh_user` stacked without fighting each other. The only per-tool variation was the Args shape + server_api method name + `run_active_response`'s ar_allowlist gate. Each handler is ~15 lines.
- **T3's security-hardening freelance.** The implementer's own invariant analysis turned up a real gap (`escape()` insufficient for XML 1.0 Char restriction) and fixed it proactively. This is exactly the pattern the subagent-patterns memory flags as valuable tier-A freelance. Saved a review-fix cycle.
- **T9's honest documentation.** The `confirm_required` vs `parse_error` mismatch was real, the subagent found it during verification, and documented what the system ACTUALLY does rather than what the spec sketched. Documentation drift caught before it shipped.
- **Plan's full-code-in-every-step discipline.** Every test, model, and handler had executable code in the plan. Subagents made adaptations (ty ignores, fixture shape, lint fixes) but the shape of the implementation never drifted from the spec. Zero NEEDS_CONTEXT dispatches.
- **MCP SDK probe produced a clean "no native support, use placeholder" decision.** T1 surfaced that toolset enablement isn't in MCP 1.27 and committed the finding inline — T7 then wired `meta={"toolset": "writes"}` with full awareness that it's descriptive, not enforcing.
- **Integration tests collected cleanly on first try.** No fixture-shape inventing beyond the M4a precedent. Ports allocated deterministically.

## Plan bugs caught during execution

1. **Plan's T3 fuzz strategy needed tightening.** The sketch's `st.text()` default alphabet emits C0 controls and unpaired surrogates — which the XML-safety validator (correctly) rejected, causing `filter_too_much` health check failures. Subagent restricted alphabet to `[0x20, 0xFFFD]` minus category `Cs` — the correct fuzz surface.
2. **Plan's `xml.sax.saxutils.escape` defense was necessary but not sufficient.** T3 implementer noticed XML 1.0 forbids most C0 controls, unpaired surrogates, and `U+FFFE`/`U+FFFF` even via numeric reference; `escape()` passes them through. Added `_reject_xml_illegal` validator — a real security fix.
3. **`confirm_required` in `SAFE_CODES` is declared but never raised.** Pydantic's `Literal[True]` check fires at Args parse and surfaces as `parse_error` before any post-parse defensive check can fire. Kept in `SAFE_CODES` for future use (if elicitation or out-of-band confirm paths land in M4c+), but operator docs describe `parse_error` as the actual observed shape.
4. **Plan's T5 constructor-signature sketch was slightly off.** `ServerApiClient` takes `user: SecretValue` + `verify_tls: bool`, not `username: str` + `verify: bool`. Subagent adapted. Symptom of plan-being-written-ahead-of-code — documented in the retro for future plans to cross-check constructor signatures against actual source.
5. **T5's "7 methods" title vs 6 body methods.** The client adds 6 Wazuh-specific methods (`isolate_agent`, `restart_agent`, `add_agent_to_group`, `remove_agent_from_group`, `upload_rule_file`, `run_active_response`) supporting 7 tools (because `create_rule` and `update_rule` both call `upload_rule_file`). Plan title was off-by-one; subagent correctly implemented 6 and flagged. Cosmetic plan bug.
6. **T9 doc drift.** `confirm_required` was in the audit/error section of the plan; reality is `parse_error`. T9 implementer caught before doc shipped.

Total: 6 plan bugs caught, half of them security-relevant (the XML-1.0-illegal codepoint gap is the biggest). M4a had 9; M4b's lower count reflects the more defined scope + more detailed plan.

## Subagent behavior patterns (delta from M4a)

- **Proactive security hardening is worth soliciting in tier-A prompts.** T3's prompt didn't explicitly ask "look for additional XML-injection surfaces" — but the subagent's own careful analysis surfaced the C0-controls gap. The prompt's "Tier-A: this is a security surface" framing signals the subagent to invest more in invariant analysis than in a mechanical implementation.
- **Ambient LSP noise categories remain stable.** Pydantic metaclass unresolved-attribute, pytest fixture unused-params, stale module-resolution on newly-added files, "Code is unreachable" false-positives on idempotency guards — all present in M4b, all benign, all CLI-clean. No new category emerged.
- **ty `missing-argument` on intentional pydantic-ValidationError tests.** When a test calls `SomeModel(required_field=missing)` to assert the `ValidationError` path, ty correctly flags it at static-analysis time. Use `# ty: ignore[missing-argument]` on that single line; the runtime test still asserts what it asserts.
- **Adapting constructor signatures to actual source.** Plans written with sketched signatures (e.g. `username: str` instead of `user: SecretValue`) get adapted correctly by subagents — they read the existing source before running tests, find the real shape, match it. Documented as M4a pattern, confirmed in M4b.
- **Fixture plumbing stays inline when M4a precedent established it.** All 3 new M4b integration fixture pairs stayed inline in `test_m4b_writes.py`; no shared conftest churn. Matches T8 prompt explicitly calling out the precedent.
- **"Tier-A with reviewer" vs "tier-A with spot-check" is a real methodology knob.** M4a invoked full dual review on all 5 tier-A tasks, 3 found Critical/Important issues. M4b invoked full dual review on 1 of 5 tier-A tasks (T3 renderer only) and controller spot-check on the others; zero issues slipped through. The knob's correct setting depends on task novelty: T3 introduced a new security primitive, T4/T6/T7 composed primitives already reviewed in M4a. When in doubt, err toward full review — but don't reflexively dispatch review on mechanical glue.

## Known gaps carried to M4c / M5

- **`write.restart_manager` tool.** `create_rule` / `update_rule` upload files but don't activate; operator restarts out-of-band. An automated restart tool is considered but deferred to M4c at earliest. Today's workflow: upload → verify file exists → `systemctl restart wazuh-manager` → confirm rule loads.
- **Formal toolset SDK wiring.** T1 probe confirmed `mcp==1.27.0` has no native toolset client-enablement. `meta={"toolset": "writes"}` descriptive placeholder; revisit in M4c once SDK lands the feature.
- **Multi-agent scope for `run_active_response`.** Single-agent per call in M4b. Batched remediation is M4c or later.
- **Per-tenant RBAC policy resolution in multi-tenant HTTP.** Still carries M4a's `TODO(M4b)` — but M4b didn't touch it. Reinforces that this is an M4c prerequisite for any enterprise multi-tenant deploy.
- **MCP elicitation-based confirm flow.** If SDK surface stabilises, M4c could replace the `Literal[True]` + description-level pattern with an elicitation roundtrip (more robust against prompt-injected Claude). Today's confirm flow is description-level + audit-grade enforcement, not cryptographic.
- **Cross-tenant write leak tests.** M5 scope — session pinned to tenant A runs every write tool, assert no tenant B data/resource touched.
- **`confirm_required` code is unused at runtime.** Pydantic's `Literal[True]` short-circuits to `parse_error`. Declared in SAFE_CODES for a future post-parse defensive check path (elicitation fallback, streaming flows). If M4c / M5 doesn't activate it, consider removing in a future milestone.

## Methodology notes for M4c and beyond

- **Plan detail investment pays at execution time.** M4b's plan had concrete Pydantic field types, test bodies, and renderer code. Zero tier-A fix-after-review cycles. M4a's plan was less concrete; 3 fix cycles. The plan writing-time investment pays off in dispatch-phase review overhead.
- **Tier-A review is not auto-dispatched.** When a tier-A task composes already-reviewed primitives (T4/T6/T7 in M4b), a controller spot-check on invariants is sufficient. When a tier-A task introduces a new security primitive (T3 renderer), full dual dispatch + review is still the right default. The "full review always" default from M4a was correct when every tier-A task was novel; M4b's mix-of-novel-and-composition made it finer-grained.
- **Probe notes compose.** M4a's T14 FastMCP list_tools probe set the precedent; M4b's T1 toolset SDK probe reused it. Both produced concrete decisions in `docs/superpowers/notes/` that the downstream implementers consumed directly. Cheap, high-value.
- **"Security freelance" in tier-A prompts.** Framing a tier-A task as "this is a security surface" with specific invariants to preserve signals the subagent to invest more in analysis. T3 is the canonical example: the XML-escape-doesn't-catch-C0-controls gap wasn't in the prompt, but the subagent found it because the prompt's framing invited careful thought.
- **Integration fixture DRY via inline pattern.** M4a established the inline-per-file pattern for new integration fixtures; M4b followed it without churn in shared conftest. This keeps the shared harness lean and each test file self-contained. Good trade-off if the fixtures diverge across milestones.
- **Ship-time discipline held.** `ruff format` alignment as its own commit (M2: `3c20a8d`, M3: `6fa3fce`, M4a: `765cd59`, M4b: `4e68838`); specific-file staging instead of `git add -A`; version bump + retro in the ship commit; tag + push with `--tags`. Consistent across four milestones now.
