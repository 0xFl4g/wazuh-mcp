# M4a Production Hardening — Retrospective

**Tag:** `v0.4.0-m4a`
**Plan:** `docs/superpowers/plans/2026-04-23-wazuh-mcp-m4a-hardening.md`
**Spec:** `docs/superpowers/specs/2026-04-23-wazuh-mcp-m4a-hardening-design.md`
**Predecessors:** M1 walking skeleton, M2 multi-tenant HTTP + OAuth, M3 read-path tool surface (`v0.3.0-m3`)

## Outcome

M4a shipped. 383 unit tests green (up from 271 at M3 ship — 112 new tests across the M4a surface). Ruff + ruff format + ty + pytest all clean. 20 integration tests collected (up from 15), 6 new for M4a behaviours — runnable on the amd64 GitHub Actions integration runner that also landed this milestone. The milestone decomposition call (M4a hardening, M4b write-tools) pays off: the hardening bundle is a coherent ship without write-tool design work crowding it.

Landed surfaces:

- **Three production `SecretStore` drivers.** `AWSSecretsManagerStore` (aioboto3 + moto tests), `VaultSecretStore` (hvac KVv2 via `asyncio.to_thread` + `hvac.Client` mocks), `SqliteAgeSecretStore` (pyrage + aiosqlite, real-key roundtrip). `CachingSecretStore(inner, ttl=300)` wrapper with TTL + single-flight coalescing and positive-only caching.
- **RBAC on both sides.** Default admin/analyst/readonly role allowlists + per-tenant override merge. `is_allowed` prefix+exact matcher exercised by `~200` hypothesis cases over generated tool names + allowlists. Both `list_tools` (UX filter) and `call_tool` (security guard) wrapped on `mcp_app._mcp_server` per T14 probe findings. Call-time denials emit audit events before raising `ToolError("Unknown tool: {name}")` (info-hiding).
- **Per-tenant + per-session rate limits.** `TokenBucket` primitive + `InProcessRateLimiter` with `asyncio.Lock`. Defaults: tenant 250/60s (leaves 50/min headroom under Wazuh's 300/min cap), session 60/60s. Fail-closed via `WazuhError("rate_limited")`.
- **OpenTelemetry + Prometheus.** One emission layer via the OTel SDK + Prometheus exporter bridge. Six metric families (`mcp_tool_calls_total`, `mcp_tool_duration_seconds`, `wazuh_upstream_errors_total`, `jwt_refresh_total`, `rate_limited_total`, `audit_dropped_total`). `mcp.tool.call` spans with `mcp.{tool,session,tenant,user,outcome}` attributes. `/metrics` HTTP endpoint; optional stdio metrics sidecar via `WAZUH_MCP_METRICS_ADDR`.
- **Pluggable audit sinks.** `MultiSinkAuditEmitter` fans out to any combination of `StderrSink` (stdio-safe default), `StdoutSink` (HTTP-only), `FileSink` (size-based rotation + keep-N), `HttpSink` (batched POSTs with exponential backoff), `WazuhIndexerSink` (bulk writes to `wazuh-mcp-audit-YYYY.MM.DD` via the existing `IndexerClientPool`). Bounded per-sink async queue with drop-oldest + interruptible backoff + `_safe_record_drop` hook wired to `audit_dropped_total`.
- **`@instrumented_tool` central chokepoint.** Every M3 tool registration routes through one decorator that composes, in order: RBAC guard → rate-limit acquire → OTel span → handler → audit emit → metric bumps. The 32 internal `audit.emit` calls from M3 tool bodies were removed; the decorator is now the sole audit-emit point per call.
- **CI.** New `integration.yml` GitHub Actions workflow (amd64 ubuntu-latest, nightly + manual-dispatch) runs the full integration suite on real Wazuh + Keycloak. Local arm64+darwin path still works: `@pytest.mark.requires_manager` auto-skips manager-dependent tests via `tests/conftest.py` predicated on `platform.system() == "Darwin" and platform.machine() == "arm64"`.
- **Operator docs.** Three new guides — `m4a-secrets.md` (driver selection + per-driver setup + caching + error mapping), `m4a-observability.md` (OTel env vars + Prom scrape config + metric families + stdio sidecar + span semantics), `m4a-audit.md` (sink selection matrix + Wazuh Dashboards index-pattern install + backpressure signals + concurrency model).

## What went well

- **Scope decomposition.** Splitting the v1-design-spec's M4 into M4a (hardening) and M4b (write-tools) was the right call. Hardening is a coherent bundle — secrets + RBAC + rate-limit + OTel + audit sinks all compose around the `@instrumented_tool` chokepoint. Write-tools get their own design cycle in M4b without being crowded by observability plumbing.
- **T14's SDK probe before T25's wiring.** Spent one dispatch discovering that FastMCP 1.27 has no native `list_tools` filter hook, documented the decision (Option (c) — wrap handlers on `_mcp_server`), pointed T25 at the note. T25's subagent followed the probe verbatim and landed invariant-pinning tests for the handler identity. No dead ends, no "how does FastMCP actually work" exploration in the middle of T25.
- **Tier-A dual reviews caught Critical bugs.** T25's review found the single most important M4a bug: `_rbac_call_tool` raised `ToolError` BEFORE dispatching to the decorator, so the security-contract rejection path for call-time denials emitted no audit events. That's the kind of thing a red-team would exploit for enumeration with no log trail. The reviewer's eye on "what's the audit event shape on EVERY error path" caught it. T24's review caught the `functools.wraps` omission that would've broken FastMCP schema introspection — invisible at unit-test level, fatal in production.
- **`@pytest.mark.requires_manager` + arm64-darwin auto-skip.** Local dev on Apple Silicon stays unblocked; integration tests that touch the Server API skip cleanly. The CI amd64 runner covers the full suite nightly. Matches the M3 retro's request ("add a skip marker OR an amd64 CI runner"); we did both and they compose.
- **`MultiSinkAuditEmitter` refactor maintained the `AuditEmitter` legacy alias.** All 17 M3 tool bodies' `from wazuh_mcp.observability.audit import AuditEmitter` import path kept working through the refactor — no tool-body signature changes, just the `audit.emit(...)` calls removed (32 sites). The downstream test fixtures were scope-creep-migrated in the right direction.
- **Hypothesis fuzz on RBAC matcher.** ~200 generated tool-name + allowlist combinations with explicit substring-bypass distractors (`alertsfoo.x`, `agents_hidden`, `alerts.*x`, empty string). Zero bypasses. Same pattern as M3's hunt_query fuzz — reusable template for any future allowlist.
- **Methodology carried through.** Brainstorm → spec → plan → subagent exec → retro matched M1/M2/M3 cadence. Tier-A solo + tier-B batch partition kept token cost reasonable on a milestone larger than M3. Combined spec+code review as default worked for T12+T13 RBAC; three tier-A tasks (T15, T24, T25) needed fix passes but the review → fix → green loop stayed tight.

## Plan bugs caught during execution

1. **Double-audit after decorator refactor.** Plan T25 said "handler bodies are unchanged; only the outer dispatch swaps" — but the existing 17 M3 tool bodies each call `audit.emit(...)` internally, and `@instrumented_tool` also emits. Shipping as-written would double every audit event. Controller caught this pre-dispatch and the T25 prompt explicitly removed the 32 inner emits and extended the decorator's `result_count` heuristic. Not a subagent bug — a plan gap.
2. **`PrometheusMetricReader(registry=...)` signature drift.** Plan assumed the reader accepts a `registry=` kwarg; opentelemetry-exporter-prometheus 0.62b0 does not. T21 subagent adapted by capturing `prometheus_client.REGISTRY` directly.
3. **moto + aiobotocore response-object incompat.** Plan didn't anticipate that moto's synchronous response object (`AWSResponse(MockRawResponse(body))`) doesn't satisfy aiobotocore's expected async interface (`raw_headers`, awaitable `.content`). T7 subagent wrote a `convert_to_response_dict` monkey-patch in the test fixture (documented in the test's module docstring) rather than contorting the driver.
4. **Plan's `_deliver_with_retry` attempt math was correct but fragile.** Reviewer noted in T15: the combination of `while attempt < max_attempts` + `if attempt >= max_attempts` after `attempt += 1` works but traces in a way a future refactor could easily break. Not fixed in this milestone (nit) but worth a `for attempt in range(max_attempts)` refactor if touched again.
5. **`asyncio.CancelledError` not caught in decorator.** The plan's `@instrumented_tool` only caught `Exception`; cancellation during handler execution left the rate-limit token consumed with no audit trail. T24 review caught it; fix added a `BaseException` branch that emits `error_code="cancelled"` before re-raising.
6. **`functools.wraps` missing.** Plan manually renamed `wrapped.__name__` but didn't apply `@wraps(handler)`. FastMCP reads `__wrapped__`, `__doc__`, `__annotations__` for schema generation — missing these would break MCP `tools/list` output at runtime. Caught in T24 review.
7. **Call-time RBAC emitted no audit.** Plan's `_rbac_call_tool` wrapper raised `ToolError` before dispatching to `@instrumented_tool`, so the forbidden-audit branch in the decorator was unreachable. Caught in T25 review (Critical).
8. **`ValidationError` audit-code regression.** M3 tool bodies emitted `error_code="parse_error"` for Pydantic validation failures. After the T25 refactor removed the internal emit, ValidationError fell through the decorator's `except Exception` and got emitted as `error_code="internal"` — a silent label drift. T25 review flagged it; fix added a specific `except ValidationError` branch in the decorator to preserve the `parse_error` label.
9. **`emitter.start()` had no rollback.** Sequential sink.start() loop with no cleanup if sink N failed — surviving sinks would then be stopped by the outer `try/finally` with `stop()` called on partially-initialized state. T25 review flagged; fix introduced explicit rollback in `start()` and `BaseExceptionGroup` collection in `stop()`.

Total: 9 plan bugs caught — most of them specifically by tier-A reviewers. M3 had 5.

## Subagent behavior patterns (delta from M3)

- **Ambient LSP noise stayed benign.** `ty` LSP diagnostics on newly-added modules (Cannot resolve module) and pytest fixture unused-param warnings continued to appear in system-reminders throughout the milestone; CLI `uv run ty check .` remained authoritative. Pattern documented in `feedback_subagent_patterns.md` still holds.
- **Pydantic metaclass unresolved-attribute LSP noise is new.** `TenantConfig.secret_prefix`, `rate_limit`, `audit_sinks` triggered LSP `unresolved-attribute` warnings on test files because the LSP doesn't fully evaluate Pydantic v2's ModelMetaclass at index time. CLI clean. Treat as a subclass of the existing ambient-LSP pattern.
- **Pyrage + native extensions.** `pyrage` has no type stubs; ty CLI handles it cleanly at the top level (the install resolves) but per-attribute access (`pyrage.x25519`, `pyrage.encrypt`, `pyrage.DecryptError`) needs per-line `# ty: ignore[unresolved-attribute]`. T9 subagent did this proactively; keep the pattern when importing other Rust-backed Python extensions.
- **Fix agents are cheap.** Three tier-A tasks (T15, T24, T25) needed review fix passes. Each fix prompt was ~2000 words (concrete WHAT/WHERE/WHY/SUGGESTED per issue + workflow + report-back template), and each fix landed in one subagent dispatch averaging ~120s wall time. Compared to M3's zero-fix rate, this session's fix overhead was significant but measurable (~600s total across three fixes for tier-A work). The alternative — skipping review — would have shipped Critical bugs.
- **`git add -A` is a footgun.** At M4a ship I used `git add -A` to stage the ruff-format commit and accidentally included the 5 `.DS_Store` files that have been in the session-start git status since day 1. Had to reset + restore staged + delete + recommit. Matches CLAUDE.md-style guidance: **stage specific files at ship time, don't batch-stage.**
- **M3's `test_bounded_queue_drops_oldest_when_full` pattern.** Calling `submit()` before `start()` is a supported pre-start buffer path now. The T15 hardening made this explicit (`_pre_start_buffer: list[dict[str, Any]]` drained at `start()`).

## Known gaps carried forward to M4b

- **Write-tool surface** — `isolate_agent`, `add_agent_to_group`/`remove_agent_from_group`, `restart_agent`, `create_rule`/`update_rule`, `run_active_response`. Each gated by `TenantConfig.write_allowlist` (still unused) + `confirm:true` arg + double-audit + role check at `list_tools` time. This is the whole M4b spec cycle.
- **Per-tenant RBAC policy resolution in multi-tenant HTTP.** `_rbac_policy` currently captures the primary tenant's `role_tool_allowlist` in a closure and ignores the `session` argument. Single-tenant HTTP + stdio work correctly; multi-tenant HTTP wires every session's policy to the primary tenant. `TODO(M4b)` comments in both `build_app` and `build_http_app`. Fix: lookup `session.tenant_id` → `TenantRegistry` → per-tenant `role_tool_allowlist`.
- **Formal MCP toolset SDK support.** `meta={"toolset": "..."}` placeholder remains; client-enabled subsets depend on the Python SDK landing an API we can wire.
- **`asyncio.ExceptionGroup` in `MultiSinkAuditEmitter.stop()`** — may need a `BaseExceptionGroup` fallback for callers that don't know about the 3.11+ type. No issue in 3.12, but future 3.10 support (if ever) would need attention.
- **Vault integration tests.** M4a unit-tests Vault against `hvac.Client` mocks only. A real dockerized Vault in the integration fixture was out of scope; revisit if a deploy surfaces a bug mocks couldn't catch.
- **`WazuhError.scope` field for rate-limit metric labels.** Today `@instrumented_tool` derives scope via `"tenant" in rle.message` substring check — brittle but works. A structured `scope` field on `WazuhError` (or a `RateLimitExceeded` subclass) would be cleaner. Tier-A review nit.
- **Audit event `outcome` vocabulary.** `cancelled` is now emitted on handler cancellation but isn't in the spec's canonical outcome list. M4b should either add it to the spec or reconsider the label.
- **`AppConfig`/`HttpAppConfig.audit_emitter` startup partial-failure observability.** If `_build_sinks` raises (bad URL, missing FileSink path, etc.), the error surfaces from the constructor with no audit-emitter-specific context. A wrapper in `build_app`/`build_http_app` that emits `RuntimeError("audit sink build failed: {driver}")` would help operators diagnose config bugs.

## Methodology notes for M4b and beyond

- **Tier-A dual review is load-bearing, especially on async lifecycle code.** M3 shipped with zero review catches beyond nits. M4a had 9 plan bugs, 3 of which were Critical or Important enough to block merge without a fix pass. The density of "async lifecycle + error paths + cross-module contracts" in M4a (audit sink drain loops, decorator error paths, server wiring RBAC+call_tool integration) is exactly the shape where human-eye review catches real bugs. For M4b (write-tools with `confirm:true` flow + double-audit + per-tool RBAC tightening), expect the same density and budget the review cycles.
- **Fix agents should get the same specificity as implementer agents.** The three T15/T24/T25 fix prompts each listed the CRITICAL/IMPORTANT issue by WHERE (file:line), WHY (1-3 sentence reasoning), and SUGGESTED (concrete patch with code). Each landed in one pass. The temptation to say "fix the review findings" broadly would probably have produced one-pass review-churn.
- **Controller-detected plan gaps save dispatch cycles.** The double-audit problem at T25 would have been caught by a reviewer, but the fix would have touched 22 files. Catching it pre-dispatch in the T25 implementer prompt saved an entire tier-A review + fix cycle.
- **Dispatches involving the existing test suite need fixture-migration guidance.** T20 (`MultiSinkAuditEmitter` refactor) migrated 14 downstream test files because the constructor signature changed from `AuditEmitter(stream=...)` to `MultiSinkAuditEmitter(sinks=[...])`. The subagent did this well but the plan should have listed each downstream file. For M4b, enumerate the test fixtures any refactor needs to touch.
- **Ship-prep has its own cadence.** At ship: `ruff format` apply (46 files this milestone — similar to `6fa3fce` at M3 ship); version bump; retro; tag; push. The sequence is mechanical but each step benefits from its own commit for clean history. Don't batch them.
- **`.DS_Store` pollution.** macOS creates `.DS_Store` files on every directory visit. They've been in the session-start git status untouched for four milestones. Adding them to `.gitignore` would prevent the `git add -A` footgun at ship time and wouldn't affect anyone else. Worth a one-line follow-on.
