## Headline

Two tenant_cfg-derived runtime concerns left primary-tenant-captured at server-build time after M4c are now correctly per-tenant:

1. **Per-tenant rate-limit budgets.** `InProcessRateLimiter.per_tenant` populated from `registry.all_tenants()` at boot. Closes the cross-tenant DOS path where a rogue session on tenant B could exhaust tenant A's bucket. Class already supported `per_tenant=`; M4d was pure wiring.
2. **Per-tenant audit-sink fan-out.** `MultiSinkAuditEmitter` dual-track refactor (`global_sinks` + `per_tenant_sinks`). `emit(session)` routes by `session.tenant_id`. Closes the cross-tenant audit leak where tenant A's events landed on tenant B's sinks. M4a sink lifecycle preserved unchanged across the refactor.

No new operator config surface. The `rate_limit:` and `audit_sinks:` blocks per tenant in `tenants.yaml` now actually apply per-tenant — schemas unchanged, behavior corrected.

## What surprised us

- **`AuditEmitter` alias compounded the migration footprint.** Plan estimate: ~25-30 `MultiSinkAuditEmitter(sinks=)` test sites. Actual: 47 of those PLUS 18 sites using the `AuditEmitter` alias (preserved by T4 as `AuditEmitter = MultiSinkAuditEmitter`) — total 65 sites across 19 files. Implementer's first test run failed with the unexpected-kwarg error, broadened the grep to catch the alias form, migrated all in one commit. Plan should have specified the broader `(MultiSinkAuditEmitter|AuditEmitter)\(sinks=` pattern. Retro lesson: when refactoring a kwarg on a class with a legacy alias, grep BOTH names at plan-write.
- **`sinks=[]` semantics shift was deliberate but undocumented in T4's commit.** Old `list(sinks) if sinks else [StderrSink()]` defaulted empty-list to stderr. New `list(global_sinks) if global_sinks is not None` keeps explicit empty empty. T5 implementer caught the test pinning the old behavior, renamed and adapted (`test_empty_sinks_defaults_to_stderr_sink` → `test_empty_sinks_is_explicit_no_op`). Strictly safer for multi-tenant deployments — operator passing `global_sinks=[]` likely means "no globals, just per-tenant", not "give me stderr by default". Worth a doc callout in T4's commit message next time a deliberate semantic shift hides inside an apparently-mechanical kwarg rename.
- **Plan-time fixture drift on sink config discriminator.** Plan referenced `StderrSinkConfig(type="stderr")` and `WazuhIndexerSinkConfig(type="wazuh_indexer", index_prefix=...)`. Real schema uses `kind:` as the discriminator (Pydantic Literal default). With `extra="forbid"` the plan-shaped calls would have ValidationError'd. T9 implementer adapted by dropping the explicit discriminator kwargs (Literal default fires automatically). Same pattern as M4c's six fixture drifts (TenantConfig fields, set_current_session import path, etc.). The pre-plan grep ruled out most of these, but `kind:` vs `type:` slipped through. Spec→plan→execute discipline keeps catching these on first dispatch.
- **`refill_per_sec=0.0` rejected by Pydantic.** Plan-shaped test config used `BucketConfig(capacity=2, refill_per_sec=0.0)` to simulate a hard cap. Real schema has `Field(gt=0.0)` so 0.0 fails parse. T2 implementer adapted to `0.001` — sub-second tests never gain a token. Same shape lesson; if config field has a positive-only validator, the plan should encode the lower bound at plan-write.
- **`_EmptyRegistry` Protocol expansion required.** Adding `all_tenants()` to the `TenantRegistry` Protocol (T1) exposed a previously-unchecked structural-typing site: the `_EmptyRegistry` fallback in `build_http_app` only had `get()`, and a `# type: ignore[assignment]` was masking the gap. T1 implementer added the new method to the fallback class to keep ty green, removed the masking ignore, and properly typed the existing `get`. Strict improvement (real type contract instead of suppression). Plan should have listed `_EmptyRegistry` as a touch site for any Protocol expansion.
- **Three trivial LSP `tenant_id` unused warnings on `_EmptyRegistry` fallback methods.** Confirmed orthogonal LSP noise per the established pattern. CLI `ty check` green throughout. Pattern unchanged from M4a/M4b/M4c retro recordings.
- **Integration nightly hadn't fired yet for M4c.** M4c shipped 13:12 UTC same-day as M4d began; nightly runs ~6-12h after push, so no validation against the 3 new M4c integration tests was possible pre-M4d. No issue blocked M4d (unit baseline was clean), but the M4c nightly green-or-not signal arrives after M4d ship. Carry forward: post-M4d push, watch the integration run for both milestones simultaneously.

## Lessons retro-applied to next milestone's plan

1. **Grep for `(ClassName|LegacyAlias)\(kwarg=` on any kwarg-rename refactor.** M4d's 18-site `AuditEmitter` alias miss is the second time this milestone something escaped the literal-class grep (`type=` vs `kind=` discriminator was the other). Broaden patterns at plan-write.
2. **Pin lower-bound validators in test config.** `Field(gt=0.0)`, `extra="forbid"`, `min_length=1` — when a plan's test fixture sets a value at the boundary, verify the validator allows it. M4c had this lesson on field shapes; M4d had it on numeric bounds.
3. **List Protocol implementors when expanding the Protocol.** `_EmptyRegistry` would have been on the touch-site list if the plan asked "every class structurally typed as TenantRegistry". Future Protocol-expansion plans should grep `Protocol.__subclasshook__`-style or `class \w+:.*get\(self, .*\) -> ` shapes pre-plan.
4. **Document deliberate semantic shifts inside apparently-mechanical kwarg renames.** T4's `sinks=[]` defaults-to-stderr → stays-empty was correct but undocumented in the commit. Future commit messages on dual-purpose refactors should call out both the mechanical change AND any silent semantic shift.
5. **Watch nightly integration runs for the milestone-pair.** M4d shipped before M4c's first nightly fired. The M4c-nightly + M4d-nightly results land together post-push.
