# M2 Multi-tenant HTTP + OAuth — Retrospective

**Tag:** `v0.2.0-m2`
**Plan:** `docs/superpowers/plans/2026-04-21-wazuh-mcp-m2-oauth-http.md`
**Predecessor:** M1 walking skeleton (tag `v0.1.0-m1`)

## Outcome

M2 shipped. 176 unit tests green (up from 98 in M1), ruff + ty clean. HTTP transport boots, returns a valid RFC 9728 metadata document, and rejects unauthenticated requests to `/mcp` with RFC 6750–compliant `WWW-Authenticate` headers. Unit + security-negatives suite exercises every auth-layer invariant from the spec. stdio path preserved bit-for-bit.

## What went well

- **The methodology from M1 paid off.** Phased-depth review (tier-A dual-review on security-critical; tier-B implementer-only) caught real bugs without burning the budget on scaffolding:
  - **JWKS cache** — dead `_fetched_at` field removed; three failure-path tests added.
  - **OAuth factory** — hardened to reject `alg=none` at construction time (closes the joserfc "operator-misconfigures-allowlist" hole); added `aud`-as-list and missing-`sub` tests.
  - **API-key store** — tightened prefix check to `$argon2id$` (blocks argon2i/argon2d downgrade); broadened exception catch so verification failures can't leak via stack trace.
  - **HTTP middleware** — fixed RFC 6750 non-compliance (`error="forbidden"` isn't a valid value; swapped to `insufficient_scope`; added `realm=` per RFC); tightened prefix match so `/mcpfoo` doesn't accidentally match `/mcp`.
- **Batching adjacent tier-B tasks in single implementer dispatches** (T2+T3+T4, T5+T6+T7, T12+T13, T16+T17, T21–T24) kept velocity up.
- **Pre-flight against real installed SDKs** (joserfc, authlib, uvicorn, FastMCP `streamable_http_app()`) during the brainstorming phase prevented most plan-vs-reality drift. The one miss (authlib 1.7 deprecating `authlib.jose`) was caught immediately and swapped to joserfc.

## Plan bugs and omissions caught during execution

1. **`_ensure_discovered` would have deadlocked** — plan's literal code acquired `self._lock` then called `_refresh` which also acquired the same non-reentrant `asyncio.Lock`. Fix: release the lock before the refresh call. Caught in Task 8.
2. **joserfc 1.6 deprecation** — authlib's `authlib.jose` is deprecated; the successor is `joserfc`. Plan updated mid-flight.
3. **Starlette lowercases header keys** — plan assumed `ctx["headers"]["Authorization"]` would work, but ASGI normalises to lowercase. Middleware now title-cases headers before handing them to factories.
4. **`app.state.protected_resource_metadata`** — wrapping Starlette in middleware made `request.app.state` point at the wrong object. Replaced with closure-over-dict.
5. **`FastMCP.streamable_http_app()` returns a Starlette app**, not a plain callable — good catch during plan-time pre-flight; no implementation drift.
6. **Keycloak 26 direct-grant flow** — users imported with `emailVerified: true` and empty `requiredActions` still hit `"Account is not fully set up"` during password grant. Root cause not fully diagnosed within the session's time budget; Task 20 integration tests deferred.

## Gaps carried forward

- **No OTel, no metrics, no rate limits** — all explicitly M4 scope, not gaps.
- **No real SecretStore backends** (AWS SM, Vault, sqlite_age) — explicitly M4 scope.

## Post-tag closure: Task 20

Landed after the `v0.2.0-m2` tag (commits `af044cf`, `94afc70`). The four OAuth end-to-end tests now run against the Keycloak 26 fixture; full integration suite is 7 green (4 OAuth + 3 pre-existing `search_alerts`).

- **Workaround taken** (cheaper than the four listed above): flip `serviceAccountsEnabled: true` on `wazuh-mcp-client` and mint tokens via `grant_type=client_credentials`. The existing `aud` and hardcoded `tenant_id` protocol mappers apply to service-account tokens, and the absent `wazuh_mcp_role` claim deliberately exercises `OAuthSessionFactory`'s `default_rbac_role` fallback path. No realm-flow hacks, no token-exchange preview feature.
- **Two real HTTP transport bugs surfaced** that would have shipped broken in M2 — `build_asgi_app` mounted FastMCP's sub-app at `/mcp`, but FastMCP already exposes its handler at `/mcp` internally, so the effective URL was `/mcp/mcp` and POSTs to the advertised endpoint 404'd. The outer Starlette also wasn't forwarding the sub-app's `lifespan_context`, so FastMCP's session-manager task group never started and every request raised `"Task group is not initialized"`. Both fixed in `af044cf`. The unit-level `test_asgi_composition` missed them because the dummy MCP app uses a different route layout and no lifespan — worth remembering for M3: **dummy sub-apps in unit tests must match the real sub-app's route and lifespan shape, or integration will surface the drift**.
- **Follow-ups noted for M3**:
  - `WWW-Authenticate` on 401 lacks the `resource_metadata=<url>` parameter expected by MCP 2025-06-18. Current value is RFC 6750–valid but spec-incomplete. Small fix when next touching `transport/http.py`.
  - `docker/seed_alerts.py` uses wall-clock `datetime.now(UTC)`; a host clock drift between seed and test (macOS sleep/resume) flips the `search_alerts` tests red. Consider fixed ISO offsets and `time_range="24h"`, or re-seeding per-test.

## Architecture survivors (unchanged through M3)

- `SessionFactory` protocol + `Config` / `OAuth` / `ApiKey` / `Chain` drivers — the indirection layer.
- `session_ctx.ContextVar` — per-request session; tool handlers pull from it.
- `IndexerClientPool` — per-tenant lazy client cache.
- HTTP transport (`SessionMiddleware`, `build_asgi_app`, RFC 9728 metadata, health/ready) — tool handlers don't change when M3 adds more tools.
- All M1 primitives (`Session`, `SecretValue`, query builder, error scrubbing) — unchanged.

## Planning guidance for M3

- **The combined spec+code reviewer dispatch worked well throughout M2.** Keep it as the default for tier-A tasks.
- **When an agent dispatch runs long** (60+ minutes), split the task smaller rather than adding an implementer-self-time-budget and hoping. (M2 had one 6M-duration dispatch on T19 that was mostly docker image pulls — fine — but next time, separate "infra setup" from "write code" into distinct dispatches.)
- **Keycloak realm exports for Keycloak 26+** need explicit `authenticationFlows` overrides, not just user-level `requiredActions: []`. Before M3 lands a new IdP fixture, validate the full flow manually against a bare Keycloak 26 container first, THEN bake it into a JSON export.
- **When a docker smoke blocks on "Account is not fully set up" or similar auth-flow-resolution errors**, time-box to 15 min. Pivot to admin-API workarounds or token injection rather than chasing the realm config.
- **The ty + ruff + pytest trifecta stayed clean for all 25 tasks.** Keep all three in CI.
