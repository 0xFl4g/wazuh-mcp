## What went well

- **Phase 1 was uniformly mechanical.** T1-T5 were single-file YAML/MD/Python authoring tasks; each implementer dispatch averaged ~50-180s wall time. The corpus + scoring + slash command all landed without surprises. Plan-detail investment (concrete YAML bodies in the plan) paid off — implementers transcribed verbatim.
- **Tier-A review caught a real plan defect.** T7's plan said both tenants would share the realm issuer and the IssuerIndex would "resolve to whichever tenant's config is found first" — but `IssuerIndex.__init__` actually raises on duplicate issuers, an invariant pinned by `test_duplicate_issuers_rejected`. The reviewer traced the impact (every integration test that boots `mcp_http_server` would fail at config-load) and proposed Path A (relax invariant to collapse-to-None) vs Path B (revert to phantom URL — also broken). Path A landed as a separate fix commit. **This is exactly the value Tier-A review is designed to catch** at the seam between primitive design and downstream integration impact.
- **Eval harness redesign was the right call.** Mid-spec we redesigned the eval from a CI pytest + `ANTHROPIC_API_KEY` to a Claude Code slash command. Result: zero recurring API cost, audit trail in git, and the corpus + scoring stay framework-agnostic so a future v1.x community contributor can wrap them in CI with their own key. Each artifact (corpus, scoring, slash command) ships as a discrete shippable piece.
- **Batching adjacent tier-B tasks worked.** T10+T11 (security CI) and T12+T13 (destructive isolation) each ran as one batched implementer dispatch producing two atomic commits. Lower coordination cost; same atomicity.
- **Implementer-side plan corrections were healthy.** T4 removed an unused `# ty: ignore`; T9 swapped mypy `# type: ignore` syntax for ty `# ty: ignore` and dropped an unnecessary `f`-string prefix (F541). Each was reported in the DONE message — exactly the DONE_WITH_CONCERNS-as-plan-correction-signal pattern from M4c/M4d retros.

## What surprised us

- **The IssuerIndex invariant.** Plan-author baseline §4 explicitly addressed the "two tenants share issuer" case and predicted IssuerIndex behavior — the prediction was wrong. The actual behavior was pinned by a unit test the plan author didn't grep for. Lesson: when a plan claims behavior at the seam between two subsystems (auth/factory + tenancy/index), grep for assertions or tests that pin the invariant in either subsystem before locking in the design. ~5 minutes of `grep -rn "duplicate_issuer\|IssuerIndex"` would have caught it.
- **Plan-time fixture drift extends to test count math.** Plan-author wrote "513 passed" as the unit-test baseline target after T9, but the real baseline grew from 506 (pre-M5a) to 514 (after T4 added 8 score tests) to 516 (after T7-fix added 2 shared-issuer tests). The numbers in plan-text drift fast when fix dispatches add tests; treat them as orienting checkpoints, not contracts.
- **T6 is genuinely a controller-only task.** It needs wazuh-mcp connected to the controller's own Claude Code session as an MCP server. We didn't have that during execution, so T6 is deferred (status `pending` in the task list, marked DEFERRED in description). The plan's Phase 3-4 work was independent of T6 so the milestone shipped without it. Lesson: when planning, mark tasks that require maintainer environment configuration explicitly as such, not just "controller-inline".

## Tier-A composition pattern

M5a is **not** a composition-only milestone — T7 introduced a novel auth primitive (Keycloak claim-mapper). The plan correctly tagged it Tier-A and full review caught a real Critical (the IssuerIndex invariant clash). This is the EXPECTED pattern from `feedback_subagent_patterns.md`: "Full review only when introducing a novel primitive."

The "Tier-A spot-check validated three milestones running" pattern (M4b/M4c/M4d) doesn't apply here — those were composition-only. M5a is the first milestone since stabilization with a true novel primitive, and the review cycle paid for itself.

## Plan-detail investment outcome

Plan-detail investment cost ~30 minutes more at plan-write time, mostly on T4 (concrete TypedDict shapes + 7 test bodies inline) and T7 (full Keycloak realm JSON in the plan body). Result:

- Zero plan-time fixture drifts on T4 — implementer transcribed and adapted minor lint/style.
- T7's Critical was an invariant clash, NOT plan-time fixture drift — plan-author had verified field names and import paths but missed the cross-subsystem invariant. Plan-detail investment caught the simpler drifts (per M4d's lesson) but doesn't substitute for cross-subsystem grep.
- T9's f-string F541 + ty syntax issues caught at implementer-side. Cheap to fix.

Net: plan detail prevents the ~30 minute dispatch-time corrections per task that M4c suffered. M5a reproduced the M4d pattern (zero plan-time fixture drift on the bulk of tasks).
