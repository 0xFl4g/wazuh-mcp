## What worked

- **Aggressive Phase 2 batching cut dispatch count.** Plan budgeted 11 separate Phase 2 dispatches; actual was 3 implementer dispatches (CI workflows batched as one, multi-manager+Vault batched as one, T-G4+T-G5a batched as one). 1700+ lines of plan content executed cleanly per batch. The implementer reports show they did the cross-task work coherently — no scope confusion from grouped tasks.
- **Tier-A combined reviewer dispatch for T-A1 only.** T-A1 was the only novel security primitive (`agent_group_allowlist` field + resolver). T-A2 was wiring, validated via T-A1's APPROVED + controller spot-check on the implementer report. Followed M3 retro's "combined reviewer halves token burn" pattern.
- **Plan-author corrections were caught by implementers, not reviewers.** Six significant plan corrections surfaced during execution:
  - `_spawn_server` location (not in conftest as plan claimed; in `test_m4b_writes.py`)
  - Active-response command must be `isolate` not `restart-wazuh` (only `isolate` registered in ossec.conf)
  - MCP client API name `streamable_http_client` (not `streamablehttp_client`)
  - `agents.list_agents` (not `agents.list`)
  - `VaultSecretStore` constructor uses `address`/two-arg `get(tenant_id, key)` (not `url`/`get(key)`)
  - wazuh-mcp loads config from a DIRECTORY via `WAZUH_MCP_CONFIG_DIR`, NOT per-credential env vars (chart updated accordingly)
  - Health endpoints `/healthz` + `/readyz` (not `/health/live` + `/health/ready`)
  - Substring-match was in `observability/decorators.py:132`, not `metrics.py`
  - `BucketConfig.refill_per_sec` validator is `gt=0` not `ge=0` (T-G1 tests adapted)
  - Spec §7.4 phantom-token Path A and Path B both non-viable (Keycloak doesn't expose realm private keys, doesn't support arbitrary tenant_id claims). Plan-author chose Path C (side-car JWKS) at plan-write; implementer further refined to thread-singleton pattern because module-scoped sync `mcp_http_server_audit_sinks` fixture can't await an async-context-manager.
- **T-G5a code-only investigation surfaced a real bug.** Implementer identified that `WazuhIndexerSink` hardcodes `index_patterns=['wazuh-mcp-audit-*']` in its template registration, which never matches the configured per-tenant `index_prefix` (`tenant-b-audit-*`, `local-audit-*`). This is the actual root cause of `test_per_tenant_audit_routing` failing — found without running the test, just by reading the lifecycle code. Fix queued for v1.0.x patch with docker validation.
- **Controller-inline Phase 5/6 saved ~6-9 dispatches.** Per M4d/M5a precedent — doc/ship phases need no implementer.

## Plan-author lessons for v1.1+

1. **`_spawn_server` lives in `test_m4b_writes.py`, not conftest.** Future fixture-spawn tasks should either inline the boilerplate (T-A3 precedent) or extract `_spawn_server` to a `tests/integration/_helpers.py` (worth doing in a v1.0.x patch).
2. **`VaultSecretStore.get(tenant_id, key)` is two-arg.** All future Vault docs/tests must use the two-arg form.
3. **wazuh-mcp's config model is directory-based, not env-var-based.** Never plan deploy artifacts assuming per-credential env vars.
4. **Health endpoints are `/healthz` + `/readyz`** — settled wire-shape, document everywhere.
5. **`WazuhIndexerSink._send_with_retry` retry-then-silently-drop is dangerous in tests.** T-G5a's investigation identified that errors are swallowed after `max_attempts`. Future audit-pipeline work should add structured failure logging or a metric that distinguishes "delivered" from "dropped after exhaustion."
6. **For phantom-token-style tests requiring trust of arbitrary issuers**, the side-car JWKS pattern (Path C) is the canonical approach. Keycloak admin API can't substitute. Document this in any v1.x test infra rework.
7. **Plan-time invariant grep extends to substring-match site discovery.** T-G1's plan grep targeted `metrics.py` and missed the actual substring-match in `decorators.py:132`. Future cross-cutting refactors should grep the broader source tree, not just the file the spec mentions.
8. **Helm chart authoring is mostly mechanical YAML** but each template needs to be cross-checked against actual app behavior (env vars, health endpoints, config-file shape). Plan templates should NOT pre-fill placeholder env vars — leave a "verify at task time" callout.
