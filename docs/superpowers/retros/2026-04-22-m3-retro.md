# M3 Read-path Tool Surface — Retrospective

**Tag:** `v0.3.0-m3`
**Plan:** `docs/superpowers/plans/2026-04-22-wazuh-mcp-m3-tool-surface.md`
**Spec:** `docs/superpowers/specs/2026-04-22-wazuh-mcp-m3-design.md`

## Outcome

M3 shipped. 271 unit tests green (up from 176 in M2) plus ~15 integration tests (not auto-run; user bootstraps the docker stack separately). Full read-path tool surface live: 17 tools (4 alerts, 5 agents, 2 vulns, 2 mitre, 2 hunt, 2 fim) + 3 resources + 3 prompts + a new Server API client + pool sibling to the indexer client. ruff + ty clean. Wazuh 4.8+ only (vulnerabilities live in the indexer under the states index; no 4.7-era Manager API fallback).

Tag applied at the end of this retro.

## What went well

- **Dispatch strategy.** 16 implementer dispatches batched per the plan's tier-index (T2+T4, T6 three-commits, T9+T11, T16+T17, T18+T19). Tier-A tasks got solo dispatches; tier-B batches halved dispatch count without losing quality signal.
- **Hypothesis fuzz on `hunt_query`** (T15) validated zero grammar bypass against ~600 generated inputs; no counterexamples found.
- **Subagents caught several real issues:**
  - T6's TestClient lifespan context requirement (`with TestClient(app) as client:` needed for lifespan events).
  - T14's "description contains script" substring-check false positive fixed via semantic dict walker.
  - T22's discovery that FastMCP's `@mcp.resource()` decorator auto-dispatches URI templates (replaced plan's manual URI pattern-match sketch).
- **Plan-to-implementation drift was small.** Most deviations were mechanical (RUF lint-suggested refactors, existing fixture names in tests differing from plan placeholders).

## Plan bugs caught during execution

1. **`aclose()` vs `aclose_all()` naming.** Plan specified `aclose()` on `ServerApiClientPool` but `IndexerClientPool` uses `aclose_all()`. Subagent flagged; renamed inline via small patch (commit `dffd400`). T22 wiring stayed simple as a result.
2. **Hunt DSL dangerous-key substring check.** Plan's `"script" not in str(q)` assertion tripped on `"rule.description"` which contains `"script"` as a substring. T14 subagent replaced with a dict-key walker; T15 fuzz uses the same pattern.
3. **FastMCP resource decorator.** Plan sketched manual URI pattern-matching for `resources/read`, but FastMCP's native `@mcp.resource(uri)` handles templates automatically. T22 subagent used the SDK-native pattern; simpler than the plan's sketch.
4. **Session priming in stdio path.** Extracting `_register_everything()` meant every tool now calls `current_session()`, so stdio wiring must prime the contextvar at startup. Added to `run_stdio()`. Not a plan bug exactly, but an emergent complication from the shared-helper refactor.
5. **`preferred_username` list handling.** Plan's OAuth `_pick_wazuh_user` took first item of a list claim; working as designed.

## Known gaps carried forward to M4

- **Write tools** (`isolate_agent`, `add_agent_to_group`, `restart_agent`, `create_rule`, `run_active_response`) — per v1 design, v2 territory. `TenantConfig.write_allowlist` remains unused in M3; M4 wires it.
- **RBAC-aware `list_tools`** — filter tools by `Session.rbac_role`. M4 scope.
- **Per-tenant + per-session rate limits** — M4.
- **OTel spans, Prometheus metrics, pluggable audit sinks** — M4.
- **Real `SecretStore` drivers** — AWS SM, Vault, sqlite_age — M4.
- **Formal MCP toolset SDK support** — `meta={"toolset": "..."}` is the placeholder; real SDK support deferred.
- **`streamablehttp_client` → `streamable_http_client` migration in production code** — tests use the new name (T6), production still uses legacy (OK — no production consumer yet).
- **Cross-tenant leak test suite** — M5 scope but worth flagging.
- **Eval harness** — M5 scope (ship-blocker for v1.0.0).

## Architecture survivors (unchanged through M3, safe to assume into M4)

- `Session` frozen dataclass (extended with `wazuh_user: str | None` — additive, not breaking).
- `SecretValue` — M1 hardening unchanged.
- `SessionFactory` protocol + all 4 drivers — unchanged.
- `IndexerClient` + `IndexerClientPool` — unchanged (M3 only adds sibling `ServerApiClient` + `ServerApiClientPool`).
- HTTP transport — unchanged except `resource_metadata=` on 401 `WWW-Authenticate` (T5) and the Mount/lifespan fix baked in pre-M3.
- `AuditEmitter` default=stderr — unchanged.
- `WazuhError` + `SAFE_CODES` — extended with `not_found` + `upstream_timeout` (additive).
- Query-builder pattern — still the single-point DSL-enforcement layer.
- Audit contract per tool — `(tool, args, outcome, result_count, duration_ms, error_code?)`.

## New architecture primitives (M3 additions, safe to assume into M4)

- `ServerApiClient` + `ServerApiClientPool` — port 55000, JWT lifecycle, `run_as` per-request.
- `_register_everything(mcp_app, indexer_pool, server_api_pool, audit_emitter)` helper — keeps stdio + HTTP paths in sync.
- `resources/` package + `TEMPLATES` registry + `make_json_content()` helper.
- `prompts/` package + `make_user_message()` helper.
- `hunt_query` grammar with `FIELD_ALLOWLIST` + `OP_ALLOWLIST` + hypothesis fuzz.

## Planning guidance for M4

- **Write tools:** start from the M3 tool-registration pattern — add a `write_ok: bool` check against `TenantConfig.write_allowlist` as the decorator's first gate.
- **RBAC-aware `list_tools`:** FastMCP may need an `add_filter` or similar; probe the SDK first.
- **OTel:** the `current_session()` contextvar is the natural attribute hook.
- **Real SecretStore drivers:** AWS SM comes first (aioboto3); the existing `YamlSecretStore` is a working reference for the protocol shape.
